## 背景・現状整理

### 収集インフラ

| 項目 | 現状 |
|------|------|
| 実行環境 | ローカル PC（Windows）のみ（GitHub Actions は IP ブロックで収集不可）|
| タスクスケジューラ | 01:30 に実行予定 |
| 収集後 PC シャットダウン | 60 秒カウントダウン付き（キャンセル可） |

### データソース

| サイト | ステータス | 備考 |
|--------|-----------|------|
| アナスロ（anaslo） | ✅ 有効 | 差枚・G数・BB/RB 取得可能 |
| みんレポ（minrepo） | ✅ 有効 | 差枚・G数・出率 取得可能。headless=False 必須（Cloudflare 対策）|
| daidata | ❌ 無効 | 差枚が取得できないため無意味 → 全店舗 disabled |

### IP ブロック対策：ローテーション設計

同一店舗について「アナスロ」または「みんレポ」を日ごとに切り替えることで、
各サイトへの短期的なリクエスト数を半減させる。

#### ローテーションルール

```
(today.day % 2) == (group_index % 2) → アナスロで取得
(today.day % 2) != (group_index % 2) → みんレポで取得
```

- 同一店舗を `store_group` で紐付け（例: `"espace-kabuki"`）
- どちらか一方で取れれば十分（同日同一店舗の二重取得は不要）
- 将来的に店舗数が増えても、各サイトへのリクエストは常に「全店舗の約半数」に抑えられる

#### stores.yml の設計例

```yaml
stores:
  - id: espace-kabuki-anaslo
    name: エスパス日拓歌舞伎町新館
    source: anaslo
    store_group: espace-kabuki
    group_index: 0          # 偶数日はアナスロ、奇数日はみんレポ
    enabled: true
    url: "https://..."

  - id: espace-kabuki-minrepo
    name: エスパス日拓歌舞伎町新館
    source: minrepo
    store_group: espace-kabuki
    group_index: 0          # 同じ group_index → ローテーションで排他
    enabled: true
    tag_url: "https://..."
```

## タスクスケジューラ & シャットダウン設計

### 実行スケジュール

| 項目 | 設定値 |
|------|--------|
| 実行時刻 | **毎日 01:30** |
| 実行スクリプト | `run_daily.ps1` |
| 実行ユーザー | ログオン状態不問（タスクスケジューラに登録）|

### `run_daily.ps1` の処理フロー

```
01:30 タスクスケジューラ起動
  │
  ├─ 1. collect（ローテーション対応）
  │       └─ Discord 通知（成功 / 失敗）
  │       └─ logs/ に記録
  │
  ├─ 2. export（JSON 生成）
  │       └─ Discord 通知（成功 / 失敗）
  │       └─ logs/ に記録
  │
  ├─ 3. git add / commit / push（docs/ の変更を GitHub へ）
  │       └─ Discord 通知（成功 / 失敗）
  │
  └─ 4. シャットダウン（エラー有無にかかわらず必ず実行）
          └─ shutdown /s /t 60 /c "60秒後にシャットダウンします。中止: shutdown /a"
             ※ 60秒以内に `shutdown /a` を実行するとキャンセル可能
             ※ エラーがあっても必ずシャットダウン（Discord + logs で後日確認）
```

### シャットダウン PowerShell 例

```powershell
# run_daily.ps1 末尾
Write-Host "全処理完了。60秒後にシャットダウンします。中止するには: shutdown /a"
shutdown /s /t 60 /c "スロットデータ収集バッチ完了 - 60秒後にシャットダウン（中止: shutdown /a）"
```

### Phase 1: source カラム追加（DB スキーマ変更）

**目標**: 各台データに「どのサイトから取得したか」を記録する

#### DB 変更内容

```sql
-- 新規テーブル
CREATE TABLE source_sites (
    id   TEXT PRIMARY KEY,   -- "anaslo", "minrepo", "teramoba"
    name TEXT NOT NULL,      -- "アナスロ", "みんレポ", "teramoba2"
    url  TEXT
);

-- DailyMachineData に source_id カラム追加
ALTER TABLE daily_machine_data ADD COLUMN source_id TEXT REFERENCES source_sites(id);
```

#### 変更ファイル

- `src/models.py`: `SourceSite` テーブル追加、`DailyMachineData` に `source_id` FK 追加
- `src/db.py` または `src/storage.py`: source_sites の初期データ挿入
- `tests/test_models.py`: SourceSite テーブルのテスト追加

### Phase 2: ローテーションロジック実装

**目標**: 日付とグループインデックスに基づいてアナスロ/みんレポを自動切替

#### stores.yml スキーマ拡張

```yaml
- id: espace-kabuki-anaslo
  store_group: espace-kabuki
  group_index: 0        # ← 追加
  source: anaslo
  enabled: true
  ...
```

#### 変更ファイル

- `config/stores.yml`: 全店舗に `store_group` / `group_index` を追加
- `src/cli.py`: `collect` コマンドにローテーションフィルタ追加
  ```python
  # 今日取得すべきソースを判定
  today_parity = date.today().day % 2
  for store in stores:
      group_index = store.get("group_index", 0)
      expected_source = "anaslo" if today_parity == group_index % 2 else "minrepo"
      if store["source"] != expected_source:
          continue  # 今日はスキップ
  ```
- `tests/test_cli.py`: ローテーションロジックのユニットテスト追加

### Phase 6: 過去データ バックフィル

**目標**: 過去の日付のデータを少しずつ取得して DB を充実させる

#### 設計

- `collect --date YYYY-MM-DD` で特定日付のデータを取得（既に実装済み）
- バックフィルスクリプト `scripts/backfill.ps1` を作成
  ```powershell
  # 例: 過去30日分を1日1回ずつ取得（サイトへの負荷を最小化）
  $dates = (1..30) | ForEach-Object { (Get-Date).AddDays(-$_).ToString("yyyy-MM-dd") }
  foreach ($d in $dates) {
      python -m src.cli collect --date $d
      Start-Sleep -Seconds 120  # 2分待機（IP ブロック対策）
  }
  ```
